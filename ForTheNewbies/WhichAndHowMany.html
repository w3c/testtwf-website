    7.7.3 The DataTransfer interface

    DataTransfer objects are used to expose the drag data store that underlies a drag-and-drop operation.

    interface DataTransfer {
  attribute DOMString dropEffect;
  attribute DOMString effectAllowed;

  readonly attribute DataTransferItemList items;

  void setDragImage(Element image, long x, long y);

  /* old interface */
  readonly attribute DOMString[] types;
  DOMString getData(DOMString format);
  void setData(DOMString format, DOMString data);
  void clearData(optional DOMString format);
  readonly attribute FileList files;
};
    dataTransfer . dropEffect [ = value ]

    Returns the kind of operation that is currently selected. If the kind of operation isn't one of those that is allowed by the effectAllowed attribute, then the operation will fail.

    Can be set, to change the selected operation.

    The possible values are "none", "copy", "link", and "move".

    ...
    dataTransfer . clearData( [ format ] )

    Removes the data of the specified formats. Removes all data if the argument is omitted.

    ...

    DataTransfer objects are used during the drag-and-drop events, and are only valid while those events are being fired.

    A DataTransfer object is associated with a drag data store while it is valid.

    The dropEffect attribute controls the drag-and-drop feedback that the user is given during a drag-and-drop operation. When the DataTransfer object is created, the dropEffect attribute is set to a string value. On getting, it must return its current value. On setting, if the new value is one of "none", "copy", "link", or "move", then the attribute's current value must be set to the new value. Other values must be ignored.

    ...

    The clearData() method must run the following steps:

    1. If the DataTransfer object is no longer associated with a drag data store, abort these steps. Nothing happens.

    2. If the drag data store's mode is not the read/write mode, abort these steps. Nothing happens.

    3. If the method was called with no arguments, remove each item in the drag data store item list whose kind is Plain Unicode string, and abort these steps.

    4. Let format be the first argument, converted to ASCII lowercase.

    5. If format equals "text", change it to "text/plain".

      If format equals "url", change it to "text/uri-list".

    6. Remove the item in the drag data store item list whose kind is Plain Unicode string and whose type string is equal to format, if there is one.

    The clearData() method does not affect whether any files were included in the drag, so the types attribute's list might still not be empty after calling clearData() (it would still contain the "Files" string if any files were included in the drag).

    ...

    Another Example

    Note that "Categories", "Context, and "Content Model" sections are mostly for authors, not user agents, as user agents need to accept everything and somehow make it work.

    However, sometimes these sections do mention a quality that user agents should implement, such as in the case of "Sectioning root" and other "section" related terms. The concept of sectioning, if implemented, would involve a user-agent algorithm, the implementation of which would then need to be tested.

    So, drill down into every term you see, to make sure to catch everything required of a user agent within this definition box.

    There are also other statements within the spec that do not use the RFC 2119 language but might still need to be tested.

    These are called "candidate requirements."

    Determining whether or not it should be tested is something that requires reading other parts of the spec.

    For this particular example, no test is necessary for the statement above, because the actual requirement for setting the value of dropEffect when the DataTransfer object is created is defined in another part of the spec.

    On the other hand, if it is a simple enough test to write, you should perhaps go ahead and include it here anyway, to ensure that the technology is truly completely tested within your specific test suite.

    In sum: if the statement is an observation about your current test target that is fully contained within another section of a spec and does not particularly affect your tests, leave it to that spec section's tester to test the statement. However, if the statement is something that your current tests rely upon being correctly implemented as their own basis, strongly consider writing the tests within your test suite, even if they may be duplicated elsewhere.

    Another example

    This part of the spec does not use much RFC 2119 language, but it is full of testable statements.

    Note that

    • Normative statements aimed at authors have been crossed out.
    • Normative statements aimed at user agents are highlighted and in bold.
    • Candidate requirements for user agents are merely highlighted.

    Also note that some of these statements will require digging further into the spec to catch all the necessary tests you should write:

    • Double-check exactly how a boolean attribute should behave
    • Find out exactly how an integer should be parsed given different kinds of inputs

    The ol element represents a list of items, where the items have been intentionally ordered, such that changing the order would change the meaning of the document.

    The items of the list are the li element child nodes of the ol element, in tree order.

    The reversed attribute is a boolean attribute. If present, it indicates that the list is a descending list (..., 3, 2, 1). If the attribute is omitted, the list is an ascending list (1, 2, 3, ...).

    The start attribute, if present, must be a valid integer giving the ordinal value of the first list item.

    If the start attribute is present, user agents must parse it as an integer, in order to determine the attribute's value. The default value, used if the attribute is missing or if the value cannot be converted to a number according to the referenced algorithm, is 1 if the element has no reversed attribute, and is the number of child li elements otherwise.

    The first item in the list has the ordinal value given by the ol element's start attribute, unless that li element has a value attribute with a value that can be successfully parsed, in which case it has the ordinal value given by that value attribute.

    Each subsequent item in the list has the ordinal value given by its value attribute, if it has one, or, if it doesn't, the ordinal value of the previous item, plus one if the reversed is absent, or minus one if it is present.

    ...

    The reversed, start, ... IDL attributes must reflect the respective content attributes of the same name. The start IDL attribute has the same default as its content attribute.
    • If the reversed attribute is present, it indicates that the list is a descending list (..., 3, 2, 1).
    • If the reversed attribute is omitted, the list is an ascending list (1, 2, 3, ...).
    • If the start attribute is present, user agents must parse it as an integer, in order to determine the attribute's value.
    • If the element has no reversed attribute, the default value of the start attribute, used if the attribute is missing or if the value cannot be converted to a number according to the referenced algorithm, is 1.
    • When the element has a reversed attribute, the default value of the start attribute, used if the attribute is missing or if the value cannot be converted to a number according to the referenced algorithm, is the number of child li elements.
    • Assuming the first li child has no successfully-parsable value attribute, the first item in the list has the ordinal value given by the ol element's start attribute.
    • If the first li child element has a value attribute with a value that can be successfully parsed, the first item in the list has the ordinal value given by that li element's value attribute.
    • Each subsequent item in the list has the ordinal value given by its own value attribute, if it has one.
    • If the reversed attribute is present on the list, then, for each subsequent item in the list that does not provide an ordinal value via its value attribute, that item's value is the ordinal value of the previous item minus one.
    • If the reversed attribute is NOT present on the list, then, for each subsequent item in the list that does not provide an ordinal value via its value attribute, that item's value is the ordinal value of the previous item PLUS one.
    • The reversed IDL attribute must reflect the reversed content attribute.
    • The start IDL attribute must reflect the start content attribute.
    • The start IDL attribute has the same default as the start content attribute.
    As for algorithms:

    Requirements are often stated within algorithms.

    Some algorithms in a spec may contain many individual statements or steps that don’t use each use RFC 2119 but that nonetheless state conformance requirements.

    Such algorithms are typically prefaced by a statement that does use RFC 2119 language, and the algorithm step that requires conformance testing is often written using an imperative statement.

    Again, a lot of digging around in other parts of the spec will be necessary when writing these tests - you will need to know all about "drag data store"s and how formally to convert a value to ASCII lowercase.

    The clearData() method must run the following steps:

    1. If the DataTransfer object is no longer associated with a drag data store, abort these steps. Nothing happens.
    2. If the drag data store's mode is not the read/write mode, abort these steps. Nothing happens.
    3. If the method was called with no arguments, remove each item in the drag data store item list whose kind is Plain Unicode string, and abort these steps.
    4. Let format be the first argument, converted to ASCII lowercase.

    5. If format equals "text", change it to "text/plain".

      If format equals "url", change it to "text/uri-list".

    6. Remove the item in the drag data store item list whose kind is Plain Unicode string and whose type string is equal to format, if there is one.
    • Nothing happens when .clearData is called on a DataTransfer object which is not associated with a drag data store
    • Nothing happens when .clearData is called on a DataTransfer object when its drag data store has a mode other than "read/write mode"
    • If dataTransferObject.clearData is called with no arguments, only remove all items in the drag data store's item list whose kind is Plain Unicode string.
    • If dataTransferObject.clearData is called with at least one argument, remove the item in the drag data store item list whose kind is Plain Unicode string and whose type string is equal to format (as described in steps 4 and 5), if there is one.

  3. Double-check that each assertion you have noted has not been dropped from the Nightly version of the spec. If an assertion has been dropped in the Nightly version, don't write a test for it -- just include a comment within your future test file that references the assertion and the fact that it was dropped from the Nightly version.

    • HTML Canvas 2D Context, Nightly
    • HTML Microdata, Nightly - DOES NOT EXIST
  4. Lastly, make sure that you'll get notified of future Nightly spec changes that might impact your particular tests by watching the Nightly spec on GitHub.
    • Source of both Nightly HTML 5 and Canvas 2D Context Specs
    • Source of Nightly HTML Microdata - DOES NOT EXIST
  5. Every note you made during this review will require at least one test to verify that the statement is true. (More on that topic in the Writing Tests section.)
  6. But then, stop! No other tests are required! Focus only on verifying that the specific assertions made in this particular spec are true. Do not add any tests that test for absence of any additional features (for example, do not test that an element has only X number of attributes as listed in the spec), as other specs, partial interfaces, and newer versions of the very same technology may also add features not listed in this particular spec.
